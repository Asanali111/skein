## How to use Skein in this project — read BEFORE you do anything else

Skein is the fastest path to context here. Concrete numbers:
- `project_briefing()` returns the project dashboard in ONE call — <50ms, ~300 tokens.
- `recall("<your task>")` returns top-K relevant fragments — <100ms, ~30 tokens per fragment.
- For comparison: reading 3 source files for the same context costs 3000+ tokens, 5+ round trips, and often misses the WHY.
- Embeddings run locally — no API key needed, your code never leaves this machine.

**Default behavior for any task on this project:**
1. Call `project_briefing()` first if you need the project overview (what's it doing, where it's at, what changed lately).
2. Call `recall("<task>")` for context relevant to your specific task.
3. Only fall back to `read_file` / `grep` for code-execution details that semantic recall can't give.
4. After any non-obvious decision, call `remember(...)` or `note_decision(...)` so other tools (and you in a future session) inherit it.

Each `recall` result carries a `quality` bucket: `high` / `medium` / `low` / `none`. If the top result is `quality=none`, Skein has nothing high-signal for that query — fall back to source. Honest no-knowledge is more useful than weak hits.

- Skein vision constraint (locked by user, 2026-05-17): Skein takes a slice of context-sharing work off the user; it MUST NOT grow the surface area the user has to learn. Prefer fewer commands doing more work automatically. New CLI commands are the thing to minimise. New MCP tools (called by agents, not users) are fine because they're part of the LLM's job. When tempted to add a user-facing command, first try: (a) make the daemon do it automatically with sensible defaults, (b) fold the output into an existing command (`skein doctor`, `skein up`, `skein tail`), (c) expose only as an MCP tool the agent can invoke from natural-language. This invalidates the original Q-05 phase 4 proposal (4 new CLI commands: boost/bury/gc/top); rewriting to auto-gc + MCP-only boost/bury + value-distribution-in-doctor.
Alternatives considered: (a) Ship phase 4 as planned with 4 CLI commands — rejected explicitly by user: "we shouldn't expand the toolkit for the user." (b) Make all of phase 4 MCP-only with no human-facing surface — partially adopted; some surfacing in existing commands is still needed for visibility. (c) Skip phase 4 entirely and rely on phase 3 telemetry alone — rejected; user-overrideable signal is sometimes necessary, just not via dedicated commands.
Rationale: Every new command is a thing the user has to learn, remember, and decide when to run. The whole product pitch ("AI tools finally remember each other's work") fails if the user has to babysit the memory layer with 4+ admin commands. The win is invisible context flow — Skein should feel like a daemon that just works, not a CLI suite that demands attention. Concrete consequence for Q-05: phase 4 collapses from 4 new commands into (1) daemon-side auto-gc on the existing stale-mark interval, (2) value distribution + recent gc count surfaced in `skein doctor`, (3) `boost`/`bury` as MCP tools the agent calls when the user says "this is important" / "this is wrong" — no CLI surface at all.
